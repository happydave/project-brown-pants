## Architectural Boundaries

Required: `/crates/sim/` (`sounding_sim`) depends only on Bevy sub-crates (`bevy_app`, `bevy_ecs`, `bevy_time`, `bevy_diagnostic`, `bevy_log`), never the `bevy` umbrella — so it is rendering-free and runs headless.
Forbidden: `/crates/sim/` pulling in rendering or windowing. Verify: `cargo tree -p sounding_sim` shows no `bevy_render`, `bevy_winit`, or `wgpu`.
Required: every change to simulation state is `Command`-driven through the executor — no input system, autopilot, or bus client mutates state directly; they emit `Command` messages. Two arms apply them: **field mutations** (warp, pause, maneuver) go through the pure `apply_command` in `/crates/sim/src/command.rs`; **structural changes** (component insert/remove, spawn/despawn — e.g. the gear swap on `Command::SetGear`, WI 508) go through a dedicated `Command`-triggered system, since they are outside `apply_command`'s `(clock, orbit)` reach. The invariant is the routing (a `Command` drives it), not the single function.
Required: external processes (`/crates/companion/`, future second screen / MCP AI) act on the simulation only via the runtime bus (`GET /telemetry`, `POST /command`). The bus injects `Command`s; it never mutates state.
Required: the Bevy Remote Protocol is compiled only under the app's `dev` feature, absent from default/release builds, and is distinct from the runtime bus.

## Dependency Chains

- `/crates/app/` depends on `/crates/sim/` + the `bevy` umbrella + `tiny_http` + `serde_json`.
- `/crates/companion/` depends on `/crates/sim/` (shared `Command`/`Telemetry` types) + `ureq` + `serde_json` + `glam`. No Bevy.
- `/crates/sim/` depends on Bevy sub-crates + `glam` + `serde` + `serde_json` (the latter for the WI 498 persistence format, in library code now — not just tests). No rendering; no `bevy` umbrella.
- `Command` and `Telemetry` (`/crates/sim/`) are the shared contract every other crate and the bus depend on — treat changes to them as public-contract changes. The `persist` format (`SavedDocument` + `FORMAT_VERSION`) is a second, **durable** versioned contract: it has its own `FORMAT_VERSION` line (independent of the crate version) and must stay backward-loadable via the two-stage `from_json` migration seam. Do not conflate it with the ephemeral bus.

## Linear Data Flow

Command lifecycle (the one to know):
1. A source emits a `Command` — keyboard (`/crates/app/src/main.rs`), the companion via `POST /command` (`/crates/app/src/bus.rs` → `drain_commands`), or a future MCP/tablet client.
2. The `Command` becomes a Bevy message.
3. `apply_command` (`/crates/sim/src/command.rs`, in `Update`) drains messages and applies them to `SimClock`/`Craft` — the sole writer of simulation state. Maneuvers apply at the current `clock.time` (impulsive, position-continuous).
4. `OrbitPlugin::advance_clock` advances simulated time by real-frame-time × warp.
5. `publish_telemetry` (`/crates/app/src/bus.rs`) reads the craft's `Orbit` at `clock.time` and writes a `Telemetry` JSON snapshot to a shared buffer each frame; when an active scene has published an `ActiveFlight` block it is attached (WI 569 — the live control tier, not just the orbit); the `tiny_http` thread serves it on `GET /telemetry`. (The on-rails orbit now runs headless behind the 3D scene; it is not the rendered craft.)

Craft derivation (Toy 5): a `VoxelCraft` (in `EditorState`, or loaded from a `SavedDocument`) is the single input; `mass_properties` and `area_curve` (`sim/voxel.rs`) derive mass/CoM/inertia and the aero cross-section from the same voxels. The editor draws them as gizmos each frame; nothing is precomputed or duplicated.

Render placement (Toy 4, retained): entities carry an f64 `WorldPlacement`; `track_anchor` sets the `FloatingOrigin` to the camera's X/Z and `rebase` derives f32 `Transform.translation` as `world − anchor`, pinning the camera to the render origin in X/Z with altitude in Y (the atmosphere consumes that directly).

Wheel contact (Toy 6): `Rover::step` queries the **analytic** `Terrain` (f64) per wheel — never the mesh — computes suspension + tyre + drag forces into a net wrench, and integrates the `ActiveBody` via `integrate_wrench`, sub-stepped at `SUBSTEP_DT`. The rover scene anchors floating-origin rendering at the rover (it sits at a large f64 world offset) and sub-steps the rover each frame.

Resource catch-up (Toy 7): a `ResourceGraph` (`sim/resource.rs`) advances by `integrate(target)` — analytic event-stepping, not tick-by-tick. Each step resolves effective per-reservoir/node net rates under the boundary constraints (empty reservoir's outflow ≤ inflow; full reservoir's inflow ≤ outflow; depleted node draws nothing), then jumps `dt` to the earliest saturation/depletion breakpoint and advances all quantities linearly. The graph holds its own last-integration `time`; nothing drives it from `SimClock` yet (embedding it in the world-save is deferred past the WI 508 hand-off).

Gear hand-off (Toy 8): a `Command::SetGear(kind)` flows through the bus/executor like any command, but is applied by `handoff::apply_gear_switch` (not `apply_command`): it reads the craft's current gear (`Craft` xor `ActiveBody`) plus its persistent `GearState`, builds the other gear's state at `clock.time` (`wake` evaluates the orbit's (r,v); `sleep` fits a conic through the active (r,v)), records the injected `HandoffDiscontinuity` into `LastHandoff`, and swaps the component. Because `wake`/`sleep` are mutual inverses over the shared (r,v) state, the swap injects ~0 jump; the `sim/handoff_discontinuity` diagnostic alarms if it ever does not.

The dive (Toy 9): `descent_step` (`sim/medium.rs`) advances an `ActiveBody` under gravity + drag + buoyancy each sub-step. It samples the one `FluidMedium` at the craft's altitude (`r − surface_radius`); `drag_force` and `buoyancy_force` read only that sample's density (+ submerged voxel volume), so vacuum/atmosphere/ocean are the same code path with different numbers — the multi-fluid thesis. Forces feed `integrate_wrench` (the rover's dissipative integrator). `DiveTriggerPlugin` auto-emits `Command::SetGear(Active)` (and drops warp to 1) when an on-rails craft drops below the entry interface. Since WI 527 **everything is SI** (`CentralBody::EARTHLIKE`): the dive scene runs one craft through orbit→handoff→`DescentPlugin` descent in the same units as the orbit gear — no per-scene unit convention.

Breakage (WI 518): `fracture` (`sim/breakage.rs`) takes a craft + its `ActiveBody` + an applied acceleration, converts the load to the craft's local frame, and runs the stress proxy (`failing_cut`) — axis-aligned cut planes where the outboard inertial load exceeds the crossing-bond strength. The weakest failing cut's bonds are severed and `connected_components` re-partitions the lattice; `split_active` then gives each fragment a momentum-conserving `ActiveBody` (`v = v_cm + ω×r`, inherited spin, re-derived inertia). One structural-change event, not per frame.

Compartments (WI 519): `compartments(craft)` (`sim/compartments.rs`) flood-fills the exterior from the expanded bounding-box border through air cells (solid voxels and closed doors block; open doors pass); the unreached interior air, partitioned into face-connected components, is the set of sealed compartments. A `CompartmentCache` recomputes only when `mark_dirty` is called (build/break/door toggle) — so a scene draws from the cache each frame and recomputes only on an event. Doors are air-topology only (they do not touch mass/aero/breakage).

Flooding (WI 520): a breached `FloodCompartment` (`sim/flooding.rs`) `step`s each active tick against the ambient `FluidSample` at the craft's depth — one ΔP-driven transient: vacuum/atmosphere → air equalises (decompression), liquid → water floods (or implodes past crush strength). `flooded_mass_properties` recomputes the craft's mass + CoM with floodwater point-masses each tick, and the scene feeds that to the `ActiveBody` (heavier → sinks; CoM offset → tilts). Real-time, bounded to active physics — it forces a drop out of warp.

Aero (WI 521): `aero.rs` derives lift, the stability moment, and transonic wave drag from the *same* voxel `area_curve` + `FluidSample` the dive uses for drag. Lift is thin-airfoil (`Cl = 2π·α`, stall) ⊥ to the flow; wave drag exists only where `sound_speed` is defined (a gas), so it is exactly zero in water/vacuum — the medium parameterises the one module. Forces apply to the active craft via `integrate_wrench`, as drag/buoyancy do in the dive.

Gliding re-entry (WI 526): `medium::glide_step` *applies* the aero forces in flight — lift + transonic wave drag added to the dive's central force, plus a lift-at-CoP restoring moment and `aero::pitch_damping_moment` as torque. A statically-stable craft (CoP aft of CoM, via a dense-nose/tapered area curve) entered at a nonzero angle of attack deflects into a glide and weathervanes to trim without tumbling; every term vanishes appropriately in vacuum/liquid (no medium-identity branch). `descent_step` stays as the ballistic reference. The glide is *transient* (a passive craft trims into its deflected path); a *sustained* trimmed cruise needs control (the autopilot thread).

## Known Gaps (not yet built)
